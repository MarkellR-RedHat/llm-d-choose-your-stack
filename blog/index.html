The Architecture Decisions You Cannot Undo Later

Deploying a large language model to production is not one decision — it is five, and they interact in ways that punish you at scale if you get them wrong early.

Model size determines your GPU topology. Traffic pattern determines your routing strategy. Hardware choice constrains your model options. Latency requirements dictate whether you can batch aggressively or need to disaggregate phases. Multi-tenancy determines whether your serving layer is a shared bus or a set of isolated lanes.

Most teams discover these dependencies the hard way: a proof-of-concept works on a single A100, but falls apart when traffic spikes, latency SLAs tighten, or the second team shows up wanting to share the cluster. The decisions compound. Reversing them under production load is expensive.

This post walks through each of the five decision dimensions, explains what happens when you get them wrong, and provides a concrete framework for making these choices up front — using llm-d, the CNCF Sandbox project for distributed LLM inference, as the reference architecture.

1. Model Size: The Decision That Constrains Everything Else

Model size is the load-bearing wall of your inference stack. Every other decision — GPU selection, parallelism strategy, batch size, routing approach — flows downstream from how many parameters you need to serve.

The tensor parallelism threshold

Models under roughly 30 billion parameters can typically fit on a single GPU with enough headroom for KV cache. Once you cross that line, you need tensor parallelism: sharding the model across multiple GPUs, synchronizing at every layer. This is not a linear cost increase. Inter-GPU communication (NVLink or PCIe) becomes a bottleneck, and your minimum deployment unit jumps from one GPU to two, four, or eight.

A 70B model in FP16 requires approximately 140 GB of VRAM just for weights — before any KV cache. On 80 GB A100s, that means at least two GPUs with tensor parallelism of 2. On H100s with the same memory, the math is the same, but the interconnect is faster. On MI300X with 192 GB HBM3, you can fit the model on a single accelerator, which changes your routing and scaling story entirely.

Quantization trade-offs

Quantization (FP8, GPTQ, AWQ) reduces memory footprint and can improve throughput, but it is not free. Quality degradation is workload-dependent: summarization tasks may tolerate INT4 quantization with minimal loss, while code generation and reasoning tasks often show measurable regression. The right answer depends on your accuracy requirements and your willingness to benchmark.

Rule of thumb: If your model fits on a single GPU without quantization, start there. If it requires tensor parallelism, quantize first and measure quality loss before adding GPUs. The operational complexity of multi-GPU serving is real, and avoiding it is worth a few percentage points on a benchmark.

In llm-d, the vLLM serving engine handles quantized model loading natively. Your Endpoint Picker Plugin (EPP) routes identically regardless of whether the backend is running FP16 or FP8 — the routing layer does not need to know about quantization.

2. Traffic Pattern: Bursty, Steady, or Agentic

Traffic pattern determines whether your routing is doing load balancing or cache optimization — and these are fundamentally different problems.

Steady throughput

Batch processing workloads — document summarization pipelines, classification jobs, offline embedding generation — produce predictable, steady traffic. Here, GPU utilization is the primary optimization target. Pack requests into batches, maximize throughput per dollar, and use simple load-aware routing to keep replicas evenly utilized.

Bursty, user-facing traffic

Interactive applications spike. A customer-facing chatbot might see 10x traffic variation within an hour. Bursty traffic requires autoscaling, which means your routing layer must handle a changing set of backend replicas gracefully. Round-robin breaks when pods are mid-startup and have cold caches. Load-aware routing helps, but it still ignores the most expensive operation: redundant prefill computation.

Agentic and multi-turn

This is where the decision gets interesting. Agentic workflows — tool-calling chains, multi-step reasoning, retrieval-augmented generation — produce requests that share long prompt prefixes. A multi-turn conversation appends each new turn to the same growing context. An agent calling tools repeatedly sends the same system prompt and prior tool results with every request.

The prefix caching opportunity: In agentic workloads, 60-90% of each request’s prompt is identical to the previous request in the same session. Without prefix-cache-aware routing, every request re-computes that prefill from scratch. With it, the KV cache is already warm on the target pod, and time to first token drops by an order of magnitude.

llm-d’s EPP tracks KV cache state across all vLLM replicas in real time. When a new request arrives, it identifies which pod holds the longest matching cached prefix and routes there — balanced against current load. This is not hash-based guessing. It is actual cache-state-aware routing, and it is the reason llm-d exists.

3. GPU Hardware: Matching Iron to Workload

GPU selection is often treated as a procurement decision, separate from inference architecture. That is a mistake. The GPU you run on changes your parallelism strategy, your maximum batch size, your memory budget for KV cache, and your cost per token.

The practical landscape

  • NVIDIA A100 (80 GB): The workhorse. Wide ecosystem support, mature tooling, readily available. Limited by HBM2e bandwidth for decode-heavy workloads.
  • NVIDIA H100 (80 GB): Higher memory bandwidth, FP8 Transformer Engine, faster NVLink. Better throughput per GPU, but the memory capacity is the same as A100 — you still need tensor parallelism for 70B+ models.
  • AMD MI300X (192 GB): The memory story is different. 192 GB of HBM3 means a 70B model fits on a single accelerator. This eliminates tensor parallelism overhead entirely for many production models. The trade-off is a smaller software ecosystem and less mature tooling.

Mixed fleet considerations

Real clusters are rarely homogeneous. You may have A100 nodes from an earlier procurement cycle and H100 nodes from a recent expansion. llm-d runs on Kubernetes, which means you can use node selectors and topology-aware scheduling to place model replicas appropriately. The EPP routes based on cache state and load, not GPU type — so a mixed fleet works as long as each replica can serve the model.

The Gateway API integration in llm-d means traffic management is declarative. You define routing policies in Kubernetes-native resources, not in vendor-specific configuration files. When you add new hardware, the routing layer adapts without reconfiguration.

4. Latency Requirements: The TTFT Dividing Line

Latency requirements split into two distinct problems, and conflating them leads to architectures that are bad at both.

Time to first token (TTFT)

TTFT is dominated by prefill — the phase where the model processes the entire input prompt. For a 4,000-token prompt on a 70B model, prefill can take 2-5 seconds depending on hardware. For real-time applications (chatbots, autocomplete, interactive agents), this is the metric that determines whether the experience feels responsive.

Tokens per second (TPS)

TPS is dominated by decode — the autoregressive phase where tokens are generated one at a time. Decode is memory-bandwidth-bound, not compute-bound. High TPS requires fast memory, not more FLOPS.

Prefill/decode disaggregation

When TTFT targets are aggressive (under 500ms for long contexts), the most effective architectural pattern is disaggregation: separate pools of GPUs for prefill and decode. Prefill pods are optimized for compute throughput — they chew through the prompt and produce KV cache entries. Decode pods are optimized for memory bandwidth — they generate tokens from warm cache as fast as the bus allows.

When to disaggregate: If your p95 TTFT target is under 500ms and your average prompt length exceeds 2,000 tokens, disaggregation will almost certainly outperform a unified serving pool. If your prompts are short and your latency budget is generous, the operational complexity of disaggregation is not worth it.

llm-d supports disaggregated prefill and decode natively. The EPP routes prefill requests to prefill-optimized pods and decode requests to decode-optimized pods, managing the handoff of KV cache state between them. This is orchestrated through standard Kubernetes primitives, not custom infrastructure.

5. Multi-Tenancy: Sharing Without Interference

Multi-tenancy is the dimension that enterprise teams underestimate most. The first model deployment is a single team running a single model. By the sixth month, three teams want to share the cluster, each with their own fine-tuned adapter.

LoRA adapter serving

LoRA adapters are lightweight fine-tuned layers that sit on top of a base model. A single vLLM instance can serve multiple LoRA adapters from the same base model, swapping adapters per request. This is dramatically more efficient than running separate model instances for each tenant — the base model weights (which consume 95%+ of GPU memory) are shared.

llm-d’s EPP supports LoRA-aware routing. When a request specifies a LoRA adapter, the EPP routes to a replica that has that adapter loaded (or can load it with minimal overhead), while still considering cache state and load. This means multi-tenant serving does not sacrifice the cache-aware routing benefits.

Isolation and resource sharing

Kubernetes namespaces and resource quotas provide the isolation boundaries. On OpenShift, you get additional security context constraints and network policies. The key architectural decision is whether tenants share model replicas (via LoRA) or get dedicated replicas behind the same gateway. The answer depends on your isolation requirements: shared replicas are more cost-efficient; dedicated replicas provide stronger performance isolation.

Putting It Together

These five dimensions are not independent sliders. Model size constrains GPU choice. Traffic pattern determines whether cache-aware routing matters. Latency targets determine whether disaggregation is worth the complexity. Multi-tenancy compounds everything.

The value of llm-d as a platform is that these capabilities compose. You can start with a single model on a single GPU with load-aware routing, then add prefix caching when you move to multi-turn workloads, add LoRA adapters when the second team shows up, and add disaggregation when your TTFT SLA tightens. The routing layer evolves with your requirements because it is built on Kubernetes-native primitives, not a monolithic inference platform.

The wrong time to make these decisions is when your proof-of-concept is already in production and the SLA is on fire. The right time is now.
